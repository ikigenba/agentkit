# Phase 30 — Fix: Anthropic replayed thinking block elides empty reasoning text (`thinking,omitempty`)
*Realizes design Decision 9 (the empty-thinking-text serialization rule and R-T06O-8SZX). Depends on Phase 23 (the field-name correction that introduced the `Thinking` field and its `json:"thinking,omitempty"` tag) and Phase 22 (the native lowering that enables thinking on Anthropic, so a `ReasoningBlock` exists to replay); the empty-string sibling of Phase 23's field-name trap and the Anthropic analogue of the OpenAI empty-`summary:[]` fix (Phase 16, R-OMKB-AY19).*

Fix a defect in the `anthropic` adapter's request build (`anthropic/anthropic.go`, `convertMessage` / the `wireBlock` struct tag) that Phase 23 introduced. Phase 23 moved the replayed reasoning text from the wrong `text` field into a new `thinking` field, tagged `Thinking string` with `json:"thinking,omitempty"`. That `omitempty` is itself a second, distinct trap: Anthropic's Messages API requires every assistant `thinking` **input** block to carry a `thinking` string and accepts an empty one (`""`) but rejects a missing one, while adaptive-thinking models (claude-opus-4-8 / claude-sonnet-4-6 with `type:"adaptive"`) return thinking blocks whose visible text is **empty by default** (display "omitted": the streamed block carries an empty `thinking` plus a real `signature`). The capture side is correct — the empty text lands in `ReasoningBlock.Summary == ""` and the signature in a non-empty `Opaque` — but on replay `convertMessage` builds `wireBlock{Type:"thinking", Thinking: b.Summary, Signature: sig}` with `Summary == ""`, and `omitempty` **drops the field**, so the block ships as `{"type":"thinking","signature":<sig>}` and Anthropic rejects the request with `400 messages.N.content.0.thinking.thinking: Field required`. Like the field-name trap it bites only on **replay**: the first turn has no prior thinking block, and the **second** turn of any reasoning-on adaptive-thinking conversation — the first to serialize the replayed block at `messages[1].content[0]` — fails. It is invisible to the Phase 23 fixture (R-TQ77-6QLK), which asserts only a **non-empty** `thinking` field and would silently elide an empty `Summary` with no assertion — the exact gap that let this through. This is the empty-string complement of Phase 23 and the Anthropic sibling of the OpenAI empty-`summary:[]`-always-present fix (Phase 16): the parse side is unaffected, the build side is wrong.

End state: the replayed thinking block always serializes a `thinking` field, equal to `""` when `Summary` is empty — present on the wire, never omitted — emitting `{"type":"thinking","thinking":"","signature":<sig>}`. Because `wireBlock` is shared across text/tool_use/tool_result/thinking, the fix must keep the field present **only for thinking blocks** (a bare removal of `omitempty` would force a noisy `"thinking":""` onto every non-thinking block); the recommended mechanism is a `wireBlock.MarshalJSON` that always includes `thinking` when `Type=="thinking"` and otherwise omits it (mirroring the OpenAI path, which forces `summary:[]` present with an interface-boxed empty slice that defeats `omitempty`). The `signature` and capture sides are untouched (R-IN0J-QMSI / R-XW08-D4YL still hold).

**Done when:** R-T06O-8SZX is covered by a clearly-named test — an Anthropic wire/replay test that builds a request from a history whose assistant `ReasoningBlock` has an empty `Summary` and a non-empty `Opaque`, asserting against the **raw request bytes** that the replayed thinking block carries a `thinking` field present and equal to `""` (presence distinguished from absence, not via an unmarshalled struct), that non-thinking blocks in the same request gain no `"thinking":""` field, and that the round-trip does not surface `400`/`ErrInvalidRequest`; R-TQ77-6QLK's existing non-empty assertion stays green; suite green. (Anthropic-specific — OpenAI/Gemini/Z.ai replay reasoning via their own input-item shapes and are unaffected.)
