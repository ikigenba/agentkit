# Phase 19 — Fix: Google adapter drops co-resident `functionCall`/text when a part carries a `thoughtSignature`
*Realizes design Decision 9 (the Gemini response-parse correction). Depends on Phase 11.*

Fix a defect in the `google` adapter's response parse (`google/google.go`, `parseParts`): a Gemini response part is treated as reasoning-only whenever it carries a `thoughtSignature` (the guard `if part.Thought || part.ThoughtSignature != ""` captures the signature into a pending `ReasoningBlock` and then `continue`s past the part's text/`functionCall` handling). But Gemini attaches `thoughtSignature` to the part that holds the model's output for a step — including the `functionCall` part itself and visible-text parts — not only to standalone `thought:true` parts. So when the model returns a `functionCall` part bearing a signature, the `continue` **drops the function call**: no `ToolUseBlock` is assembled, `hasToolUse` is false, `Finish` stays `FinishStop`, the tool loop never runs, and the turn ends with a text-less reasoning block — the consumer sees only token usage and the message separator, no tool call. The same guard misclassifies a real text answer carrying a signature as a `ReasoningDelta` (rendered as a thinking summary, never stored as a `TextBlock`, so history also degrades). Observed via agentrepl (`-c provider=google -c model=gemini-2.5-flash`): a clean first text turn, then "use the bash tool to count the files" and every subsequent answer turn produce only usage lines with no visible output and no tool execution. The existing fixtures (R-C8UE-VJ67 / R-IPGC-I69W) place the signature on a *separate* thought part preceding the `functionCall`, so the parser's positional-binding path works in tests while the real-API shape — signature on the live `functionCall`/text part — was never exercised; that is the gap that let this through. End state: a part is reasoning content **iff** `thought:true`; a `thoughtSignature` is captured into `ReasoningBlock.Opaque` (bound to the co-resident or following `ToolUseBlock` via `BoundToID`) independently of, and without consuming, the same part's text or `functionCall`, which are still emitted as `TextDelta`/`TextBlock` and `ToolUseBlock`. Golden-SSE replay fixtures pin both the signature-on-`functionCall` and signature-on-text shapes so the regression cannot silently return.

**Done when:** R-DNS8-QC6Z (signature-on-`functionCall` part assembles the `ToolUseBlock` and reports `FinishToolUse`), R-DRFX-VNF2 (signature-on-text part yields assistant-visible `TextDelta`/`TextBlock`, not reasoning), and R-DTVQ-N6WG (the co-resident signature still populates a non-empty `BoundToID`-bound `ReasoningBlock.Opaque` without consuming the tool call/text) are each covered by a clearly-named test; suite green.
