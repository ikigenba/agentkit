# Decision 9 â€” Package architecture & the provider adapter seam (SPI)

**Decision.** The root `agentkit` package owns the consumer types, the orchestration, and the `Provider` SPI; each provider lives in a sub-package that imports root and implements the SPI. Because a sub-package cannot implement another package's unexported methods, **the SPI is exported** (correcting Decision 5's "unexported methods" sketch). The dependency graph has no cycles: root never imports a sub-package.

```
agentkit (root)  â”€â”€ consumer types + orchestration + Provider SPI
   â–˛   â–˛   â–˛   â–˛
   â””â”€â”€â”€â”´â”€â”€â”€â”´â”€â”€â”€â”´â”€â”€ agentkit/{anthropic,google,openai,zai} import root, implement Provider
```

A consumer imports root + whichever sub-package(s) they use â€” the `database/sql` + `database/sql/driver` pattern, minus string-registration (explicit construction).

```go
// Provider is implemented by provider sub-packages. Consumers obtain one from a
// sub-package's New and assign it to Conversation.Provider; they do not call
// RoundTrip directly. This is the SPI for adding a provider.
type Provider interface {
    // RoundTrip performs ONE model call and returns a low-level stream the
    // orchestrator drains. The auto-tool-loop, history, and transparency live
    // in the orchestrator above this â€” not in the adapter.
    RoundTrip(ctx context.Context, req *Request) *RoundTrip
    Name() string // labels Error.Provider
    Pricing(model string) (Pricing, bool) // registry lookup; false = provider doesn't know this model id (validity gate, Decision 16)
}

// Request is one round-trip's input, built by the orchestrator from the
// Conversation. The adapter translates it to its wire format.
type Request struct {
    Model    string
    System   string
    Messages []Message // full history, resent every round-trip (stateless)
    Tools    []Tool
    Gen      GenSettings
}

// RoundTrip is one model call's low-level result. The adapter consumes the
// provider's incremental SSE stream on the backend, assembles the full assistant
// message (text + tool-call JSON + reasoning blocks) centrally, and exposes that
// assembled Message plus metadata. There is no delta surface: the SSE decode
// terminates in the adapter (delivery to the consumer is message-granular,
// Decision 2), so RoundTrip carries the whole assembled result, not a fragment
// iterator.
type RoundTrip struct { /* unexported */ }
func (r *RoundTrip) Message() Message // assembled assistant message (text+tool_use+reasoning blocks)
func (r *RoundTrip) Finish() FinishReason
func (r *RoundTrip) Usage() Usage
func (r *RoundTrip) Warnings() []Warning
func (r *RoundTrip) Err() error

// FinishReason is the normalized reason a round-trip ended.
type FinishReason int
const (
    FinishStop          FinishReason = iota // natural end
    FinishToolUse                            // model requested tools
    FinishMaxTokens
    FinishContentFilter
    FinishOther
)
```

Settled choices:
- **The SPI is exported** (`Provider`, `Request`, `RoundTrip`, `FinishReason`) â€” the consequence of sub-package adapters; documented as "implemented by sub-packages; not called by consumers." (Revised Decision 5.)
- **Providers may expose *optional* capability interfaces, discovered by type assertion.** Beyond the required `Provider` methods, a provider adapter may implement additional narrow interfaces that the core type-asserts (`p, ok := c.Provider.(SomeCapability)`) â€” the idiomatic Go pattern (cf. `http.Flusher`). The core never branches on `Provider.Name()` to vary behavior; provider-specific knowledge lives behind such an interface in the adapter. The first instance is `ToolSchemaTranslator` (D22).
- **Adapter does one round-trip; orchestrator owns the loop.** The adapter only translates `Request`â†’wire, opens the provider's SSE stream, parses it, **assembles partial tool-call JSON (and partial text) centrally** (absorbing fragment-vs-whole), and exposes the assembled `Message` + metadata. No tool loop, history, or `ToolResult` in the adapter, and **no consumer-facing delta surface** â€” the adapter fully consumes (and closes) the round-trip's SSE body to build `Message()` before returning it.
- **Streamed tool-call input is assembled from the argument fragments alone.** For providers that stream tool arguments incrementally (Anthropic `input_json_delta`, the OpenAI-family `arguments` deltas), the opening frame (`content_block_start` / its equivalent) carries only an **empty `{}` placeholder**, not real arguments. That placeholder must **not** be prepended to the fragment buffer: doing so concatenates to invalid JSON such as `{}{"command":"â€¦"}`, which fails the assembled-input parse and surfaces as `ErrInvalidRequest: invalid â€¦ tool input JSON` on the *first* tool call of a turn. The assembled `Input` equals exactly the concatenation of the streamed argument fragments; an absent or empty fragment stream assembles to `{}`.
- **Loop continuation keys off the assembled `Message` containing `ToolUseBlock`s**; `FinishReason` is carried for diagnostics and to map `FinishContentFilter`â†’`ErrContentFilter`.
- **`RoundTrip` exposes no event/delta surface** â€” only the assembled `Message()` and metadata. `MessageDone`/`ToolUse`/`ToolResult` are derived and emitted by the orchestrator (Decision 10) from that assembled `Message`, keeping transparency logic in one place. (The prior SPI had a `RoundTrip.Events()` yielding `TextDelta`/`ReasoningDelta`; it was removed when consumer delivery became message-granular â€” Decision 2.)
- **Reasoning-block drop-on-switch is localized in the adapter**: each adapter emits back only `ReasoningBlock`s whose `Opaque` is its own format and drops foreign ones â€” a switch sheds prior reasoning with no origin-tag on the block.
- **Each adapter lowers the native `ReasoningValue` to its own wire fields, keyed on the model's `spec.Kind` (D6/D16) â€” no shared `applyReasoning`, no cross-model normalization.** The wire shapes do not unify, so the lowering is per-provider, and a value whose kind the model doesn't accept warns+defaults at the build-time choke point (D6). The full table:

  Three load-bearing rules: (1) **Anthropic is two-axis â€” a `Level` also enables thinking** (`type:"adaptive"`); the consumer's single `ReasoningValue` means "think this hard," so an effort level implies thinking-on, and `DisableReasoning()` is the only path to the off axis (this also makes Â§7.2 thinking-block preservation engage exactly when reasoning is on). (2) **Gemini never sends both `thinkingBudget` and `thinkingLevel`** (400) â€” `spec.Kind` selects exactly one field per model; the other kind warns. (3) **The off-form is per-model** (`type:"disabled"` / `effort:"none"` / `thinkingBudget:0`), which is why `DisableReasoning()` is a distinct intent and a can't-disable model warns instead.
- **OpenAI Responses reasoning replay is mandatory, fixed adapter behavior.** The `openai` adapter sets `store:false` and injects `include:["reasoning.encrypted_content"]` on **every** request â€” never a consumer knob. Without `include`, OpenAI returns no `encrypted_content`, so the stateless multi-turn tool loop (history resent every round-trip) loses its reasoning chain ("reasoning item not found" / silent degradation on the v1 reasoning-model targets). The returned `encrypted_content` is what populates `ReasoningBlock.Opaque` for OpenAI; the resend-history model replays it verbatim. `store:false` is inseparable from `include` here â€” the ZDR path only returns `encrypted_content` when not server-stored â€” and it also keeps the adapter stateless/symmetric (no `previous_response_id`, per Decision 5). The other three reasoning providers likewise capture a non-empty `Opaque` from their own echo field (Anthropic `signature`, Gemini `thoughtSignature`, Z.ai `reasoning_content`).
- **A replayed OpenAI reasoning item must always carry `summary`, even empty.** OpenAI's Responses API requires every `reasoning` **input** item to include a `summary` array â€” it may be empty (`[]`) but must be **present**. The adapter does not request summaries (it sends only `reasoning.effort`, never `reasoning.summary:"auto"`), so `ReasoningBlock.Summary` is almost always `""`; the serialized reasoning item must therefore still emit `"summary": []` rather than omit the field. Omitting it â€” the trap being a struct tag's `omitempty`, which drops a nil/empty slice â€” makes OpenAI reject the request with `400 Missing required parameter: 'input[N].summary'`. The failure surfaces on the **second** turn of any reasoning conversation (the first turn to replay a prior reasoning item, e.g. the replayed item lands at `input[1]`), not the first, so it is invisible to single-turn tests. The serialization is exactly: `[{"type":"summary_text","text":<Summary>}]` when `Summary` is non-empty, `[]` otherwise â€” and the empty-`[]` case must survive on the wire. This is independent of any future `summary:"auto"` request knob: even with auto-summaries some reasoning items return empty and must still replay `summary:[]`.
- **A replayed tool call serializes `arguments` as a JSON string, not a nested object (OpenAI Responses).** The Responses `function_call` **input** item's `arguments` field is a **string** carrying the JSON text â€” exactly the form the model emits on output, where `arguments` accumulates as a string and the adapter's *parse* side already types it `string`. The canonical `ToolUseBlock.Input` is a `json.RawMessage` (a JSON value), so the adapter must emit it as `string(Input)` (`"{\"path\":\"PING\"}"`), never as the raw object (`{"path":"PING"}`). The trap is typing the outbound field `json.RawMessage`, which marshals verbatim as an object; it must be a `string`. Sending the object trips `400 Invalid type for 'input[N].arguments': expected a string, but got an object instead`. Like the `summary` case, this bites only on **replay**: the model's own tool call in the first round-trip succeeds, but the follow-up round-trip of the same turn â€” which resends the assistant `ToolUseBlock`s as `function_call` items to fetch the final answer â€” is the first to serialize `arguments` outbound (the replayed call lands at `input[1]`). It was previously masked by the `summary` 400, which failed validation earlier in the same request. **The same object-vs-string defect exists independently in the shared `internal/openaicompat` Chat-Completions path** that backs `zai` (and any future compat provider): there the assistant `tool_calls[].function.arguments` field is likewise typed `json.RawMessage` and set from `ToolUseBlock.Input`, so it serializes as an object while the parse side (`toolFunctionDelta.Arguments`) already types it `string` â€” the identical outbound/inbound asymmetry. It is masked on Z.ai's default base URL (`api/paas/v4`, which tolerates an object) and only surfaces once `zai.base_url` points at the strict coding endpoint (`api/coding/paas/v4`), which rejects it with `400 Invalid API parameter (type=1210)` on the same replay round-trip. Both adapters need the outbound field retyped `string` and set to `string(Input)`. Anthropic/Gemini take structured tool input directly and are unaffected.
- **A Gemini `thoughtSignature` is part-level replay metadata, not a reasoning-content marker â€” it must never suppress the visible text or `functionCall` riding on the same part.** Gemini attaches `thoughtSignature` to the part that carries the model's output for a step: a thought part (`thought:true`), a visible-text part, *or* the `functionCall` part itself (and, with parallel calls, each `functionCall` part). The Google response parse must therefore decide "is this reasoning content?" from the **`thought:true` flag alone**, never from the mere presence of `thoughtSignature`. Treating any signature-bearing part as reasoning-only and skipping the rest of it has two failure modes: it **drops the part's `functionCall`** (no `ToolUseBlock` is assembled, `Finish` stays `FinishStop`, the tool loop never runs, and the turn ends with an empty assistant message â€” only a text-less reasoning block), and it **reclassifies a real text answer as reasoning content** (the answer is folded into the reasoning summary instead of the assistant's visible text and never lands in `History` as a `TextBlock`, so context also degrades each turn). Both were observed via agentrepl on `gemini-2.5-flash`: a clean first text turn, then every tool-eliciting or answer turn produces only token usage and the message-boundary separator, with no visible output and no tool call. End state: a `thoughtSignature` is captured into `ReasoningBlock.Opaque` (bound to the co-resident or following `ToolUseBlock` via `BoundToID`, the positional rule of R-IPGC-I69W) **independently** of â€” and without consuming â€” the same part's text or `functionCall`, which still populate the assembled `Message` as a visible `TextBlock` and a `ToolUseBlock` respectively; `thought:true` parts remain the only source of the reasoning summary. This is the Gemini analogue of the OpenAI replay-shape traps above â€” a parse asymmetry invisible to any fixture that places the signature on a standalone thought part rather than on the live `functionCall`/text part.
- **A replayed Gemini `thoughtSignature` is a sibling field of the `Part`, not a field of `functionCall`.** Gemini's wire schema places `thoughtSignature` at the **part** level â€” alongside `text`/`functionCall`/`functionResponse` â€” never inside the `functionCall` object (whose only fields are `name`, `args`, and `id`). The adapter's own *parse* side already models this correctly (`thoughtSignature` lives on the `part` struct, sibling of `functionCall`), but the *request build* (`partsFromMessage`) nests the replayed signature **inside** the `functionCall` map (`call["thoughtSignature"] = sig`) rather than on the part wrapper that holds it. Gemini rejects the request with `400 INVALID_ARGUMENT: Invalid JSON payload received. Unknown name "thoughtSignature" at 'contents[N].parts[0].function_call': Cannot find field.` This is the **outbound** sibling of the inbound parse defect above and shares its outbound/inbound asymmetry shape with the OpenAI/`openaicompat` `arguments` traps (Phases 17â€“18): the parse side is right, the build side is wrong. It is **unmasked by the Phase 19 parse fix** â€” before that fix the signature on a live `functionCall` part was dropped on parse, so no `ReasoningBlock.Opaque` existed to replay and `partsFromMessage` never added a `thoughtSignature` at all; once Phase 19 began capturing the signature into `Opaque` (bound to its `ToolUseBlock`), the next round-trip became the first to serialize it outbound, in the wrong location. Like the OpenAI replay traps it bites only on **replay**: the model's own first tool-call round-trip succeeds (nothing replayed yet), and the follow-up round-trip that resends the assistant `ToolUseBlock` â€” carrying its bound signature â€” is the first to ship a misplaced `thoughtSignature` (observed at `contents[3]`, the second assistant turn, via agentrepl on `gemini-2.5-flash`). End state: the replayed signature is emitted as a **part-level** key (`{"functionCall":{â€¦}, "thoughtSignature":<sig>}`), the `functionCall` object carrying only `name`/`args`/`id`, mirroring the parse-side `part` shape. Anthropic/OpenAI/Z.ai are unaffected (no part-level signature field).
- **A replayed Anthropic thinking block serializes its reasoning text in the `thinking` field, not `text`.** Anthropic's Messages API requires every assistant `thinking` **input** block to carry a `thinking` string â€” the reasoning text the `signature` signs â€” shaped `{"type":"thinking","thinking":<text>,"signature":<sig>}`. The `anthropic` adapter's request build (`convertMessage`) lowers a replayed `ReasoningBlock` to `wireBlock{Type:"thinking", Text: b.Summary, Signature: sig}`, placing the reasoning text in the block's `text` field â€” and `wireBlock` has **no** `thinking` field at all â€” so the block ships as `{"type":"thinking","text":<text>,"signature":<sig>}`, missing the required `thinking` field. Anthropic rejects the request with `400 messages.N.content.0.thinking.thinking: Field required`. Like the OpenAI/Gemini replay traps above this bites only on **replay**: the first turn has no prior thinking block to resend and succeeds; the **second** turn of any reasoning-enabled Anthropic conversation is the first to serialize a `thinking` input block (the replayed block lands at `messages[1].content[0]`, the reasoning block emitted before the assistant's text), and fails. Because an Anthropic `Level` also enables thinking (`type:"adaptive"`, the two-axis rule above) and a `Budget` enables it directly, the trap fires for every reasoning-on Anthropic model â€” opus-4-8 / sonnet-4-6 and haiku-4-5 alike. Observed via agentrepl on `claude-haiku-4-5` with `-c thinking_budget=2048`: a clean first reply, then `error â€ş anthropic: messages.1.content.0.thinking.thinking: Field required (status=400 type=invalid_request_error)` on the next turn. The `signature` itself round-trips correctly (captured into `ReasoningBlock.Opaque`, R-IN0J-QMSI / R-XW08-D4YL); the gap is purely the outbound **field name** the reasoning text lands in, which no existing wire-shape test asserts â€” the byte-for-byte `Opaque` round-trip checks the signature, not the JSON field the summary serializes into. This is the Anthropic sibling of the OpenAI summary/`arguments` and Gemini `thoughtSignature` replay traps: the parse side is unaffected (Anthropic streams the `thinking` summary in via `thinking_delta`, not the `text` field), the build side is wrong. End state: `wireBlock` gains a `thinking` field and the `ReasoningBlock` case sets it from `b.Summary` rather than `Text: b.Summary`, emitting `{"type":"thinking","thinking":<text>,"signature":<sig>}`. OpenAI/Gemini/Z.ai are unaffected (their reasoning replay uses their own input-item shapes).
- **A replayed Anthropic thinking block must serialize the `thinking` field present even when the reasoning text is empty (`"thinking":""`), never omitted.** This is the empty-string sibling of the field-name trap directly above, and the exact analogue of the OpenAI `summary:[]`-always-present rule (the `summary` settled choice above / R-OMKB-AY19): Anthropic's Messages API requires every assistant `thinking` **input** block to carry a `thinking` string, and it accepts an empty one (`""`) but rejects a missing one. Adaptive-thinking models (opus-4-8 / sonnet-4-6 with `type:"adaptive"`) return thinking blocks whose visible text is **empty by default** (display "omitted": the streamed block carries an empty `thinking` plus a real `signature`), so the captured `ReasoningBlock` has `Summary == ""` and a non-empty `Opaque`. The trap is the **shared `wireBlock` struct tag** `json:"thinking,omitempty"`: `omitempty` drops the field when `Summary` is `""`, so the block ships as `{"type":"thinking","signature":<sig>}` â€” missing the required field â€” and Anthropic rejects the request with `400 messages.N.content.0.thinking.thinking: Field required`. Like the field-name trap it bites only on **replay** (the first turn has no prior thinking block; the **second** turn of a reasoning-on adaptive conversation is the first to serialize the replayed block at `messages[1].content[0]`), and it is invisible to the field-name fixture (R-TQ77-6QLK), which exercises only a **non-empty** thinking text â€” an empty `Summary` would be silently elided there with no assertion. `wireBlock` is shared across text/tool_use/tool_result/thinking, so simply removing `omitempty` would force a noisy `"thinking":""` onto every non-thinking block; the field must instead be emitted present **only for thinking blocks** â€” the mechanism is to type the field a pointer (`Thinking *string \`json:"thinking,omitempty"\``) set to a non-nil `&""` for thinking blocks and left nil elsewhere â€” `omitempty` drops only the nil pointer, so a non-nil pointer to `""` still serializes `"thinking":""` (the OpenAI path achieves the same end with an interface-boxed empty slice that likewise defeats `omitempty`, the `summary` settled choice above). The `signature` and capture sides are unaffected (the streamed empty `thinking` is already captured into `Summary`, and `Opaque` carries the signature â€” R-IN0J-QMSI / R-XW08-D4YL hold). End state: the rebuilt request body's replayed thinking block always carries a `thinking` field, equal to `""` when `Summary` is empty, and the second turn of an adaptive-thinking conversation is accepted (no `400`).
- **An adapter's outbound block-translation switch fails loud on an unknown block type.** Each adapter lowers the closed set of consumer `Block` types to its provider wire shape via a type switch (`anthropic.convertMessage`, the `openai` input-item builder in `partsFromMessage`/`appendBlocks`, `google.partsFromMessage`). The `Block` set is a **sealed, library-internal interface** (D3) â€” a new block type is a deliberate addition to *this* module, never external input â€” so a `case` that someone forgets to add is a programming error, not a runtime condition to tolerate. Each translation switch therefore ends in a `default` that **panics** with a descriptive `unknown block type %T` message rather than silently dropping the block (the prior behavior: an unhandled type fell through the switch and vanished from the wire request, corrupting the turn with no signal â€” exactly the silent-drop failure mode the replay-shape traps above produce on the wire). Panic, not a returned `error`: two of the three switches sit in functions with no error channel (`partsFromMessage` returns `[]map[string]any`), the condition is unreachable for any correct caller, and crashing loudly at the seam is the project's stance on an internal invariant (validate at boundaries, trust internal code; crash over silent corruption). The non-translating cost/prefix estimators (e.g. anthropic `stablePrefixTokens`) are exempt â€” they only approximate token counts for the cache-prefix heuristic, so an unrecognized block harmlessly contributes nothing and must not crash a live turn.

**Rejected.**
- *Adapters in the root package* â€” compiles every provider's deps into every consumer; loses Decision 5 isolation.
- *A separate `agentkit/driver` package for neutral types* â€” would split or duplicate `Message`/`Block`; one shared root type is simpler since both sides use identical neutral types.
- *Adapter returns the consumer `*Stream`* â€” conflates one round-trip with a whole (multi-round-trip) turn; `*RoundTrip` is the right granularity.
- *Tagging `ReasoningBlock` with its origin provider* â€” unnecessary; format-recognition achieves the drop without a consumer-visible field.
