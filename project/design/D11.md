# Decision 11 — Retry & backoff policy

**Decision.** AgentKit owns one cross-provider retry *policy*, configured on `Conversation.Retry` and applied around each `RoundTrip`. Wrapped SDKs have their built-in retry disabled so this is the single policy. This Decision owns the policy *contract* — the retryable category set, the per-round-trip budget scope, and the no-retry-after-first-byte streaming rule; the shared backoff *mechanism* that executes it (the `internal/retry` leaf package and its `Do` executor) is **D21**. Every retrying call site — the chat tool loop, MCP discovery (D17), and both embedding adapters (D19) — drives that one executor.

```go
// RetryPolicy controls automatic retrying of transient failures. The zero
// value uses defaults. Set on Conversation.Retry. The budget below is per
// round-trip (one model call), not per turn — a turn's many tool round-trips
// each get their own budget; overall turn wall-clock is the consumer's ctx.
type RetryPolicy struct {
    MaxAttempts      int           // total attempts incl. the first; 0 → default 4
    BaseDelay        time.Duration // 0 → default (e.g. 500ms)
    MaxDelay         time.Duration // backoff cap; 0 → default (e.g. 30s)
    MaxElapsed       time.Duration // overall budget across attempts; 0 → no cap
    IgnoreRetryAfter bool          // default false → honor server Retry-After / RetryInfo
}

// On Conversation:
Retry RetryPolicy
```

Settled choices:
- **Retry lives in the orchestrator, around `RoundTrip`** — uniform across all four providers; wrapped SDKs' built-in retry disabled (Google's doesn't retry anyway).
- **Retryable set is fixed** to Decision 7 categories: `ErrRateLimited`, `ErrOverloaded`, `ErrServerError`, `ErrTimeout`, `ErrNetwork`. Everything else is never retried.
- **The round-trip is the retry unit, enforced by construction** (the streaming-idempotency rule): retry eligibility is scoped per `RoundTrip`, not per turn, and is decided **solely from that round-trip's terminal error** (category + the per-round-trip budget). Because delivery is message-granular (Decision 2) — the adapter assembles each round-trip's whole `Message` on the backend and the orchestrator forwards events to the consumer `Stream` *only after* the round-trip returns error-free — a **failed round-trip delivers no events at all** (pinned by R-CDQ0-EM4Z). Retrying therefore re-issues only the current round-trip's call (same `Request`, idempotent) and re-streams it from scratch with no double-delivery, **regardless of whether earlier round-trips in the same turn already delivered events**. The "no retry once a byte has reached the consumer" guarantee is thus *structural*, not a runtime byte-counter: whole-round-trip buffering makes the byte-delivered-then-failed state unrepresentable rather than guarding against it. (The pre-Decision-2 design streamed deltas and tracked a live "first byte forwarded" flag per round-trip; message-granular delivery replaced that flag with the buffering invariant.)
- **Retry budget is per round-trip.** `MaxAttempts`/`MaxDelay`/`MaxElapsed` bound the retries of a single model call, not the whole turn — a long agentic turn legitimately makes many round-trips, each with its own budget. The overall turn wall-clock is bounded by the consumer's `context.Context` deadline (already respected by backoff waits), not by the retry policy: retry budget and turn timeout stay separate concerns.
- **Server signal first, then backoff.** Honor `Error.RetryAfter` when set (adapter-extracted per D7); otherwise full-jitter exponential backoff capped at `MaxDelay`, bounded overall by `MaxElapsed`. `IgnoreRetryAfter` disables honoring.
- **Context-aware** — backoff waits respect `ctx`; cancellation ends retrying with the context error.
- **Injectable clock** for tests to assert attempt counts and delays without real sleeps — the single `retry.Clock` defined by D21 (the chat orchestrator also reads `Now()` off it for log timestamps).
- **Configurable vs fixed**: configurable = attempts, base/cap/elapsed delays (per round-trip), honor-Retry-After toggle. Fixed = the full-jitter algorithm, the retryable/non-retryable lists, the per-round-trip scoping of the budget, and the round-trip-as-retry-unit streaming-idempotency rule.
- **MCP discovery is retried; MCP `tools/call` is not** (Decision 17, research line 440). `initialize` and `tools/list` are idempotent/read-only, so they retry under this same policy (network/timeout/5xx/429 → full-jitter backoff) but fail-fast on `401`/`403`/`400` and non-MCP `4xx`. A `tools/call` is treated as a non-idempotent POST: a transport-level failure (`ErrNetwork`/`ErrTimeout`/`ErrServerError`/`429`) is surfaced to the caller **without** automatic retry (the model may re-issue), and the same whole-result buffering applies to a tool-result SSE stream: the internal client assembles the entire JSON-RPC response (whether it arrives as `application/json` or `text/event-stream`) before returning, so no partial tool-result ever reaches the loop and there is nothing to retry mid-stream.

**Rejected.**
- *Retry inside each adapter* — four divergent policies; must be uniform and own the first-byte rule centrally.
- *Leaning on SDKs' built-in retry* — three different policies, none aware of the streaming-idempotency rule.
- *Retrying mid-stream by buffering/replaying* — can't un-deliver events already given to the consumer.
- *Equal-/no-jitter backoff* — full jitter is the researched choice.

**Verification.**
- R-P3LQ-QY2X — a retryable-category failure before any event is delivered is retried up to `MaxAttempts`, then surfaces the final error; with an injected clock the attempt count and honored delays are asserted.
- R-P4TN-4PTM — a non-retryable-category failure is never retried (asserted across the non-retryable categories).
- R-P61J-IHKB — the round-trip is the unit of retry: when a round-trip fails (even one that assembled partial content before erroring), its output is discarded and the entire round-trip is re-issued from the same idempotent `Request` and re-streamed from scratch, so the consumer never observes a partial or double-delivered message. (A failed round-trip delivers no events — Decision 2, R-CDQ0-EM4Z — so the "byte already delivered, now decide whether to retry" state is unrepresentable by construction; this is the falsifiable form of the streaming-idempotency rule at the message-granular seam.)
- R-Y878-6UDJ — a retryable-category failure on a *later* round-trip of a multi-round-trip turn, occurring before that round-trip delivers any event, is still retried: the retry budget (attempts and backoff) is per round-trip, not per turn, so each round-trip's budget is independent of earlier ones. Eligibility depends only on the round-trip's error category and its own budget, never on what earlier round-trips delivered — and since a failed round-trip delivers nothing (Decision 2), there is no cross-round-trip delivery state to consult.
- R-P79F-W9B0 — when `Error.RetryAfter` is present it is honored as the delay; when absent, full-jitter exponential backoff capped at `MaxDelay` is used; `IgnoreRetryAfter=true` ignores the server delay.
- R-P8HC-A11P — context cancellation during a backoff wait ends retrying promptly with the context error.
- R-6XDZ-VW6L — MCP discovery (`initialize`/`tools/list`) retries transient transport failures under this policy but fails fast on `401`/`403`/`400` and non-MCP `4xx`.
- R-6YLW-9NXA — an MCP `tools/call` transport failure is surfaced without any automatic retry (non-idempotent POST); its result is assembled whole before being fed back — the internal client reads the complete JSON-RPC response (whether `application/json` or `text/event-stream`) before returning — so no partial tool-result ever reaches the loop and there is nothing to retry mid-stream.
