# Decision 22 — Faithful tool-schema translation (provider-driven, no provider-name dispatch)

**Decision.** AgentKit is a **faithful translator** between the consumer's tool contract and each provider's wire dialect: a tool's JSON-Schema input is carried to the provider **preserving its meaning**, even when the provider's schema dialect is a strict subset of JSON Schema. A provider whose dialect is narrower declares that narrowness through an **optional capability interface**, `ToolSchemaTranslator`, that the orchestration core discovers by type assertion — never by branching on `Provider.Name()`. The provider translates what it can (inlining `$ref`, mapping `oneOf` onto a supported union) and reports only the **irreducible residue** — the constructs it cannot faithfully represent. The core surfaces that residue as a loud, non-fatal warning and still runs the turn; it never silently drops a construct and proceeds as if the schema were conveyed in full.

This supersedes the earlier "declare which keywords you silently drop" model: dropping `$ref`/`oneOf`/`additionalProperties` wholesale (the old `convertSchema` + `UnsupportedSchemaKeywords` behavior) weakened the consumer's contract — a stripped `oneOf` or `additionalProperties:false` lets the model emit tool input the real tool never agreed to accept. Translation preserves the contract where the dialect allows; the warning is reserved for the genuinely-untranslatable remainder, and fires **uniformly for custom and MCP tools** (previously MCP-only).

```go
package agentkit

// ToolSchemaTranslator is an OPTIONAL capability interface a Provider may
// implement when its wire schema dialect is a strict subset of JSON Schema. The
// orchestrator discovers it by type assertion (like http.Flusher); a Provider
// that does not implement it is assumed to carry tool schemas with full
// fidelity. It carries no provider name — the core never special-cases a
// provider by identity.
type ToolSchemaTranslator interface {
    // UntranslatableSchemaConstructs returns, in deterministic (sorted) order,
    // the JSON-Schema constructs this provider CANNOT faithfully represent in
    // its dialect AFTER best-effort translation of the given tool input schema
    // (e.g. after inlining $ref and mapping oneOf onto a supported union).
    // Empty or nil means the schema translates with full fidelity. The provider
    // applies the SAME translation when it builds the wire request, so the
    // residue reported here and the schema actually sent never drift.
    UntranslatableSchemaConstructs(schema json.RawMessage) []string
}
```

Settled choices:
- **Faithful translation is the default; the warning is the exception.** The provider transforms the consumer's schema into its dialect preserving meaning, and only the constructs with no faithful representation are reported as residue. A schema that uses only translatable constructs produces **no** residue and **no** warning.
- **Optional interface, discovered by type assertion — the core knows no provider by name.** `toolSchemaWarnings` (generalized from `mcpSchemaWarnings`) becomes: `t, ok := c.Provider.(ToolSchemaTranslator); if !ok { return nil }`, then for each tool it calls `t.UntranslatableSchemaConstructs(schema)` and, when non-empty, assembles the warning. Both the provider name and the schema knowledge stay out of root.
- **The warning is uniform across custom and MCP tools.** Residue is checked for every tool in the assembled set at the `Send`/discovery boundary — custom (`NewTool`/`RawTool`) and MCP alike. Attribution is `<server>.<originalName>` for an MCP tool and the tool's `Name()` for a custom tool. (Previously only MCP tools were checked, so a custom tool's schema could be weakened silently — that gap is closed.)
- **One Setting, `"tool_schema"`.** Because the warning now spans all tool sources, its `Setting` is `"tool_schema"`, not the MCP-specific `"mcp_schema"` the old MCP-only path emitted. Root owns the `Warning`, the `WarnToolSchemaLossy` code (which fires only when residue is non-empty — i.e. translation was genuinely lossy), and the attribution, building `Warning{Setting:"tool_schema", Code:WarnToolSchemaLossy, Detail:"<tool> could not be faithfully translated for <provider>; unconveyed: <construct…>"}`.
- **The `google` adapter owns the translation and is its single source of truth.** `convertSchema` performs the faithful transforms — **inline** a non-recursive `$ref` from `$defs`/`definitions` in place, and **map** `oneOf` to the dialect's `anyOf` (which Gemini accepts) — and `UntranslatableSchemaConstructs` reports exactly what that same translation cannot carry: a constraining `additionalProperties` (e.g. `false`, a closed object the dialect cannot express) and a **recursive** `$ref` (which cannot be inlined finitely). Both derive from one internal translate routine so the schema sent and the residue reported can never drift.
- **Anthropic/OpenAI/Z.ai do not implement `ToolSchemaTranslator`** (they accept JSON Schema with these constructs), so they produce no schema warnings — structurally (interface absent), not by name-exclusion.
- **D17 and D04 are rewritten in place** to describe faithful translation + residue warnings rather than silent dropping; **D9** still notes that providers may expose optional capability interfaces discovered by type assertion (this is the first instance).

**Rejected.**
- *Silently dropping unsupported keywords (the prior `convertSchema`/`UnsupportedSchemaKeywords` model)* — weakens the consumer's tool contract without conveying meaning: a dropped `oneOf` or `additionalProperties:false` lets the model produce inputs the tool rejects. A translator preserves the contract; only the irreducible residue is surfaced. This is the model this Decision replaces.
- *Hard-failing (`ErrInvalidConfig`) when a construct can't be represented* — too strict: it would block an otherwise-usable tool over a constraint the model will usually respect anyway, and pushes a translation problem AgentKit exists to absorb back onto the consumer. The promise is warn-loudly-and-proceed, not refuse.
- *Branching on `Provider.Name() == "google"`* — provider identity dispatched in a provider-agnostic core; silently wrong for any future narrow-dialect provider, and duplicates keyword knowledge root-side where it drifts from the adapter.
- *Returning the translated schema across the interface for the core to forward* — the core would then own dialect-specific schema shapes; keeping translation entirely inside the adapter (the interface reports only residue) keeps root provider-agnostic, and the adapter applies the same translation when it builds the request.
- *Having `RoundTrip` return the warnings (the macro-eval sketch)* — the adapter sees `[]Tool`, not the `mcpTool` wrapper, so it cannot attribute `<server>.<originalName>`; it would also re-warn on every round-trip and move surfacing off the discovery boundary. The optional-interface seam preserves attribution and timing.

**Verification.**
- R-SKVI-TSZQ — a `Provider` implementing `ToolSchemaTranslator` whose `UntranslatableSchemaConstructs` returns a non-empty residue for an attached tool's schema causes a `WarnToolSchemaLossy` warning at the `Send`/discovery boundary that names those constructs and attributes the originating tool; the warning is produced for a test provider **whose `Name()` is not `"google"`**, proving the core dispatches on the interface, not the provider name (a `Name()=="google"` implementation fails this).
- R-SNBB-LCH4 — a `Provider` that does **not** implement `ToolSchemaTranslator` produces no schema warnings for the same attached tool (the warning slice is empty), and the orchestration core contains no branch on a provider name for this behavior.
- R-SOJ7-Z47T — the `google` adapter's `UntranslatableSchemaConstructs` returns the **empty** set for a schema whose only narrowing constructs are a non-recursive `$ref`/`$defs` and a `oneOf` (because the adapter translates both faithfully), and returns **exactly** the irreducible residue — a constraining `additionalProperties` and a recursive `$ref` — for a schema using those; an implementation that still reports `$ref`/`oneOf` as unsupported (the old drop-list) fails this.
- R-9QWF-E6VI — the `google` adapter's tool-schema conversion **inlines** a non-recursive `$ref` (resolving it against `$defs`/`definitions`) so the produced provider schema contains the referenced definition's shape in place — the reference is preserved by substitution, not dropped — and the conversion reports no residue for it (a converter that emits a `$ref`-less schema missing the referenced properties fails this).
- R-9S4B-RYM7 — the `google` adapter's tool-schema conversion **maps** a `oneOf` (top-level or nested) to the dialect's `anyOf`, carrying each branch subschema (each itself converted), so the union is preserved rather than dropped, and reports no residue for it.
- R-9TC8-5QCW — a **custom** (non-MCP) tool whose schema contains an untranslatable construct produces the same `WarnToolSchemaLossy` warning (`Setting:"tool_schema"`) attributed to the tool's `Name()`, proving the residue warning spans custom tools, not MCP tools only (a build that warns only for MCP tools fails this).
- R-9UK4-JI3L — **[substrate: live Gemini, `//go:build integration` + key-presence skip]** a real Google/Gemini call carrying a tool whose JSON Schema uses a `$ref` and a `oneOf` is **accepted** by the live API and the tool call round-trips to a completed result — proving the translated schema is one Gemini actually accepts (not merely shaped correctly against a fake); the asserted outcome is a completed live tool-using turn, not a configured value.
