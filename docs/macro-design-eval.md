# AgentKit Macro-Design Evaluation — Module/Package Seams

## 1. Executive Verdict

**No major refactor is warranted. The macro structure is already right.** AgentKit's seams — a flat root `agentkit` package owning neutral types + orchestration + two SPIs, provider sub-packages (`anthropic`/`openai`/`zai`/`google`) that import root and implement the SPI, and an `internal/` tier of transport rails (`httpx`, `sse`, `openaicompat`, `mcp`) — are the idiomatic `database/sql` + `database/sql/driver` shape, are verifiably acyclic, and are correctly motivated by the design docs (D5, D9, D18–D20). Every "big" structural proposal that came out of the lens review (split the SPI into an `agentkit/driver` package; collapse chat and embeddings into one capability abstraction; add a typed provider-option escape-hatch seam now) was tested adversarially and **did not survive** — each either rested on a factual error, was pure code relocation that unlocks nothing, or was speculative-future work the project's YAGNI weighting explicitly says to defer. What *did* survive is one concrete, present, doc-contradicting defect: **the retry/backoff engine is duplicated four times** (root, `google`, `internal/openaicompat`, and the MCP discovery loop), even though D19 claims embeddings "reuse the D11 executor." That is a maintenance-hygiene fix, not a macro reshaping. The recommendation is therefore: **keep the seams; de-duplicate the retry engine; fix the one provider-name string leak if the MCP path is touched anyway.**

## 2. Current Macro Structure (as it actually is)

Verified against the code, not the summary:

- **Root package `agentkit`** owns three concern-groups in one namespace (105 exported identifiers):
  - **Consumer surface** — `Conversation`/`Send` → `*Stream` (`stream.go`, `orchestration.go:156-244`); `Embedder`/`Embed` → `*EmbedResult` (`embed.go`); the sealed `Block` union (`block.go:24`, sealed via unexported `isBlock()`); the sealed `Tool` with `NewTool[In]`/`RawTool` (`tool.go`); `GenSettings` (`gen.go`); `ReasoningValue` (`reasoning.go`).
  - **Orchestration** — the tool loop, history rollback, retry, JSONL logging, MCP tool-sourcing (`orchestration.go`, `mcp.go`, `retry.go`, `log.go`).
  - **Two SPIs** — chat `Provider` with `RoundTrip(ctx, *Request) *RoundTrip` (`orchestration.go:32`) and `EmbeddingProvider` (`embedding.go:16`). Result types (`RoundTrip`, `EmbedRoundTrip`) have **unexported fields** and exported constructors (`NewRoundTrip`, `NewEmbedRoundTrip`) + getter methods, so adapters build them across the package boundary but cannot reach orchestration state.
- **Provider sub-packages** import root and implement the SPI; root never imports a sub-package (confirmed). `zai` is a ~213-line shim over `internal/openaicompat`.
- **`internal/`** holds `httpx`, `sse`, `openaicompat` (shared OpenAI-compatible Chat-Completions path), and `mcp`.
- **Chat and embeddings are deliberately parallel** near-duplicate hierarchies (D18/D19/D20): separate request/result/usage/pricing types; embeddings have **no orchestrator** (the adapter owns chunk + normalize + reassemble + retry).
- **Reasoning is the one true escape hatch** (`ReasoningValue`: `Level`/`Budget`/`DisableReasoning`, a native value flowed untranslated and lowered per-adapter with warn+default). Every other provider-specific knob is either fully automatic (Anthropic prompt caching is a `char/4` heuristic at `anthropic.go:344`) or absent (no `tool_choice` on the consumer surface).

## 3. What Is Already Right (do NOT churn)

These are confirmed in code and unanimously endorsed across lenses:

- **The acyclic dependency graph with one shared neutral-type set.** Root never imports a sub-package; only `orchestration.go`/`mcp.go` touch `internal/` (both for `internal/mcp`); consumer-type files import nothing internal. This is the textbook shape and the core decision that must not change.
- **Two distinct SPIs, not one widened `Provider`.** An embed call has no `System`/`Tools`/`History`/`Gen` and returns no streamed message; one fat interface would force dead methods on each side. Anthropic/Z.ai legitimately have no embeddings. Keep `Provider` and `EmbeddingProvider` separate.
- **Embeddings having no mirror orchestrator.** There is no tool loop or history to own; the adapter owning chunk/normalize/reassemble is correct (D19).
- **`RoundTrip` granularity** — one model call, the loop above it. The opaque result type with exported constructor is exactly the right seam.
- **Sealed unions** (`Block`/`Tool`/`Event` via unexported methods) with the outbound translation switch panicking on unknown block types — the correct "internal invariant, crash loudly" stance. These depend on the types living in one package with the loop, which is itself a reason *not* to split the package.
- **`internal/openaicompat` as real reuse**, not premature abstraction (`zai` is a genuine thin shim).
- **Message-granular delivery** (adapter drains SSE, assembles the whole `Message`, no consumer delta stream) — keeps the SPI a simple synchronous `*RoundTrip` return. Not challenged; should not be.
- **The per-provider reasoning/replay lowering (D9 table) localized in adapters.** The wire-shape divergence (Anthropic `thinking`, OpenAI `summary:[]`, Gemini part-level `thoughtSignature`) is irreducible provider reality; pushing it into adapters proves the SPI is doing its job — it is correctly-localized complexity, not a seam leak.
- **Money math (pricing/usage/Cost) owned once in root** and shared by both paths.

## 4. The Macro Question: Is This the Best Design? Where Should the Seams Be?

**Yes, with one mechanism-level exception.** The lenses converged on three candidate macro moves; all three fail under scrutiny:

- **Split the SPI into `agentkit/driver`?** The intuition (clean the consumer namespace, complete the `database/sql/driver` precedent) is appealing but the proposal is *factually broken*: it classifies `Warning`/`WarningCode` as implement-only, but they are returned to consumers via `Stream.Warnings()` (`stream.go:65`) and `EmbedResult.Warnings` (`embed.go:89`), and `ReasoningInspector`/`EmbeddingInspector` are consumer introspection types (D16). Moving them to `driver` forces consumers to import `driver` — collapsing the proposal's own premise. Worse, the root orchestrator *constructs* `&Request{...}` every turn (`orchestration.go:305`) and stores `Provider` as a public field, so a `driver` package holding those types creates a root↔driver import cycle unless you also move the neutral data types — which is exactly the heavyweight move D9 already rejected (`D09.md`: "would split or duplicate Message/Block; one shared root type is simpler"). The "don't call `RoundTrip` directly" boundary is *already* enforced by Go's type system (unexported fields). The split relocates ~68 type decls plus ~108 adapter + ~61 test references for godoc tidiness that unlocks nothing.
- **Collapse chat + embeddings into one capability/modality abstraction?** Textbook over-engineering for two concrete modalities, and it fails the stated YAGNI weighting. The request shapes share nothing real. The "near-duplicate hierarchy" is shallow — the only load-bearing duplication is the retry mechanism (item below), not the type hierarchy.
- **Add a typed per-request provider-option seam now?** The design (a sealed neutral `ProviderOption` carrier, constructed only by sub-package functions, generalizing the `ReasoningValue` idiom) is *correct* — but it is speculative. Every named payoff (Anthropic cache TTL, OpenAI `response_format`/`service_tier`, Gemini `safetySettings`) is absent and unrequested in code and tests. An empty carrier every adapter type-asserts and warn-drops is pure relocation of the current absence until each adapter actually implements each option. It is backward-compatible to add a field later, so it is the canonical *cheap-to-add-later* case the brief says to defer. (The one near-universal knob, `GenSettings.ToolChoice`, could ship on its own merits without any of the carrier machinery — noted as a future small addition, not a macro change.)

**The one real present defect** is mechanism duplication, not a misplaced seam: the jittered-exponential-backoff-with-`MaxElapsed`-budget engine and its 5-sentinel retryable classifier exist **four times**, byte-for-byte identical in logic:
- root, unexported (`orchestration.go:404-447` + `retry.go:27-71`),
- `google/google.go:944-1033` (`embeddingRetry*`/`boundedEmbeddingRetryDelay`/`embeddingBackoffCap`/`realEmbeddingRetryClock`),
- `internal/openaicompat/embedding.go:100,271-353` (the identical loop + clock),
- and a fourth hand-rolled copy in `mcp.go:151-176` `discoverMCPTools`.

D19 (lines 51, 56, 62) explicitly claims embeddings "reuse … the D11 `RetryPolicy` executor." **They do not** — root's executor is unexported, so each adapter re-typed it. This is a doc-vs-code lie on the single most correctness- and cost-sensitive policy in the library, and the one concrete artifact the parallel hierarchy actually produced.

## 5. Recommendations (survivors only, ranked by unlock ÷ cost)

### R1 — De-duplicate the retry/backoff engine into one shared unit (DO THIS)
*Survived adversarial review as a hygiene fix; both refuters conceded it is a genuine, present defect worth fixing while there are still few copies.*

- **Target structure:** a single shared retry executor — `Do(ctx, policy, clock, isRetryable, attemptFn)` — plus one `Clock` interface, one `realClock`, and the one retryable-sentinel classifier. Most naturally a new `internal/retry` package (peer of `internal/httpx`/`internal/sse`). `RetryPolicy` and the D7 sentinels stay root-public.
- **Where the seam sits:** unchanged at the SPI/consumer level — this is below the SPI. Root's `roundTripWithRetry` and `mcp.go`'s `discoverMCPTools` call the shared executor; both embedding adapters delete their copies and call it per chunk (adapters keep chunking/transport/codec ownership). The two SPIs and the `Conversation`/`Embedder` surfaces are untouched.
- **Migration sketch:** lift the existing root logic verbatim; parameterize the result via a generic `Do[T]` or a result-carrying closure (chat returns `*RoundTrip`, embeddings return `([][]float32, EmbeddingUsage, error)`); unify the three test fakes (`fakeRetryClock`, `fakeGoogleEmbeddingClock`, the openaicompat fake) onto one `Clock`. **Watch the one real snag:** root's `retryClock` is dual-purpose — `log.go:47-48` uses `.Now()` for log timestamps — so keep the logging clock concern separate from the retry-only abstraction (don't fold logging into `internal/retry`). `internal/retry` must import root for `RetryPolicy`/`*Error`/the sentinels; that is the one internal→root edge, acceptable because the classifier genuinely cannot be a pure leaf.
- **Case FOR:** retry is the library's correctness/cost crown jewel and currently has four implementations that can silently drift (a `Retry-After` or `MaxElapsed` fix in one will not reach the others); collapsing them to one source makes D19's existing written guarantee actually true and gives any future single-shot modality correct retry for free.
- **Case AGAINST (honest):** it unlocks nothing *structural* — no new provider/modality becomes addable that wasn't already; the four copies are currently identical so the drift is latent, not observed; and the churn lands on the most safety-critical, most heavily-tested code. It is hygiene, not a seam change. Land it carefully as a contained internal cleanup.
- **Migration cost:** medium. **Unlock:** medium (kills a real drift liability + makes the docs honest). **Net: worth doing now, before a 5th adapter clones a 5th copy.**

### R2 — Replace the hard-coded `"google"` string in MCP schema warnings (DO THIS ONLY IF/WHEN THE MCP PATH IS TOUCHED)
*Survived as a real-but-small layering leak; both refuters agreed the smell is genuine while noting the proposed cure is mis-scoped.*

- **The leak (confirmed):** `mcp.go:179` does `c.Provider.Name() != "google"` to decide whether to emit JSON-Schema-degradation warnings — provider identity flowing up into the orchestration core, contradicting the provider-agnostic premise (and `Name()` is documented as labeling `Error.Provider`, not for behavioral dispatch).
- **The honest fix (bigger than "swap in an interface"):** the keyword knowledge itself lives in root (`unsupportedGeminiSchemaKeywords`/`collectUnsupportedSchemaKeywords`, `mcp.go:360-390`) and duplicates what `google`'s `convertSchema` already strips. Merely hiding the *name* behind a capability interface while leaving the keyword list in root does **not** decouple — it just relocates the same coupling with more ceremony. True decoupling means moving the degradation reporting to where the knowledge lives: have the `google` adapter report its schema-degradation (it already returns `[]Warning` from `RoundTrip`), so root carries zero provider-specific schema knowledge and the name check disappears as a side effect.
- **Case FOR:** removes the one provider-identity leak in an otherwise-clean core; eliminates the root↔google keyword-list drift.
- **Case AGAINST:** the only failure mode of the current code is a *missing diagnostic warning* for a hypothetical second schema-limited provider — not a correctness bug (Gemini still strips correctly regardless). Speculative future. Don't do it as a standalone task; fold it into any future change that touches the MCP schema path.
- **Migration cost:** small. **Unlock:** low. **Net: opportunistic, not standalone.**

## 5a. Rejected Proposals (considered and declined)

- **`agentkit/driver` SPI package split** — REJECTED. Mis-classifies consumer-facing `Warning`/`WarningCode`/inspectors as implement-only; creates a root↔driver import cycle because the orchestrator constructs `Request`/stores `Provider`; the "don't call this" boundary is already type-system-enforced via unexported fields; D9 and the comparative baseline (langchaingo keeps its seam in-package) both point the other way. Pure relocation, ~169 edits, zero unlock.
- **SPI file-split (`spi.go`) as a cheaper alternative** — REJECTED. Go's compilation/godoc unit is the package, not the file; moving symbols within `agentkit` changes zero references, zero godoc, zero behavior. Its own rationale is incoherent (cites `isBlock`/`isTool` co-location, but those seal at package scope and already live in separate files) and it mis-files the consumer-facing `Event` taxonomy under "SPI." An editor-affordance tweak, not a structural change.
- **Chat/embeddings capability-modality unification** — REJECTED. Speculative over-abstraction for two concrete modalities; the only real duplication is the retry mechanism (handled by R1), not the type hierarchy.
- **Typed provider-option escape-hatch seam now** — REJECTED *as a present action* (the design is correct for the future). Every payoff is absent/unrequested; backward-compatible to add later; the empty carrier unlocks nothing until adapters implement options. This is the textbook cheap-to-add-later case. (Ship `GenSettings.ToolChoice` standalone if/when wanted — it needs none of the carrier machinery.)
- **Shared build/parse replay-symmetry test contract (`internal/providertest`)** — REJECTED. Its premise ("the build side is unguarded") is factually false: every D9 replay trap already has a dedicated, R-id-tagged turn-2 regression test (`anthropic_test.go:237,300`; `openai_test.go:196,365`; `zai_test.go:154`; `google_test.go:328`) that drives the real `Send` path against an httptest server and asserts raw request bytes. The proposed `AssertReplaySymmetry(buildFn, …)` abstraction *cannot exist* — each adapter's build path returns a different unexported wire type — and asserting on a returned struct rather than marshalled bytes would *miss* the `omitempty` presence-vs-empty bug class the existing byte-level tests catch. Lower fidelity at higher cost. The residual value (a checklist for the next provider author) is a one-line doc note in D9, not a new package.

## 6. The Single Biggest Structural Change

**There is none warranted.** The most honest, evidence-grounded answer to "what is the single biggest structural change that most unlocks the future" is: *don't make one — the seams are already in the right places.* The acyclic root + sub-package + `internal/` shape, the two-SPI separation, the opaque-`RoundTrip` granularity, the sealed unions, and message-granular assembly are all correct and should be preserved. The only change worth doing is **R1: collapse the four retry copies into one shared executor** — and that is a mechanism de-duplication that fixes a documented inconsistency, not a re-drawing of any seam. Doing it early (before a fifth adapter clones a fifth copy) is the right call, but it is hygiene, not architecture.

## 7. Maintainer Goals — Direct Assessment

1. **Best available macro design / where the seams go.** Yes — at or above the idiomatic Go baseline; the `database/sql` + `driver` shape with one shared neutral-type root is correct and acyclic. No seam needs relocating. The only sub-seam-level defect is the duplicated retry engine (R1).
2. **Idiomatic Go any programmer grasps instantly.** Met. One struct + one verb + one result (`Conversation.Send → *Stream`), bare struct-literal construction, sealed unions, file-by-noun layout. The 105-identifier root namespace mixing consumer and SPI types is the one minor wart, but it is enforced cleanly by unexported fields and matches the closest peer (langchaingo); splitting it costs more than it returns.
3. **Maintainable by humans and AI agents.** Met. File-by-noun + `docs/design/DNN.md` with an INDEX mapping every Decision and R-id to a file is a strong agent-navigation aid. R1 *improves* this by making the docs honest (D19's retry-reuse claim becomes true). R2 removes a provider-name surprise.
4. **Future providers/models cheap.** Met. A new chat provider implements one interface and never touches retry (root owns it). A new OpenAI-compatible provider reuses `internal/openaicompat` (`zai` proves it at ~213 lines). R1 makes a new *embedding* provider cheaper still (call the shared executor instead of cloning ~80 lines). Adding a future modality is "one more interface in root" — equally cheap with or without the rejected `driver` split.
5. **Consistent-superset interface / escape-hatch tension.** The *consistent* half is met excellently. The *superset* half is met in exactly one place (`ReasoningValue`) and is a genuine, acknowledged gap (no `tool_choice`, opaque uncontrollable Anthropic caching, no general per-request passthrough). The *correct* fix is the typed `ProviderOption` seam that generalizes `ReasoningValue` — but it is **deliberately deferred** per goal-7 YAGNI: it is backward-compatible to add and unlocks nothing until a real consumer needs a specific knob. When that need lands, adopt that typed/sealed/sub-package-constructed design wholesale rather than a `map[string]any` blob or an LCD gateway reshape. Until then, "no escape-hatch refactor now" is the right call.
6. **YAGNI re future modalities.** Honored throughout. Speech/image-gen/rerank/more-embeddings are speculative; the structure already makes them cheap to add (one more interface, and R1 hands them correct retry for free). No modality machinery is built now. "No major refactor needed" is the evidence-backed conclusion, not a failure to find one.
