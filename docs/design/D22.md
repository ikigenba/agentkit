# Decision 22 — Provider-driven tool-schema limits (no provider-name dispatch in the core)

**Decision.** A provider declares which JSON-Schema keywords it silently drops from tool schemas through an **optional capability interface**, `ToolSchemaLimiter`, that the orchestration core discovers by type assertion — never by branching on `Provider.Name()`. This removes the one provider-identity leak in an otherwise provider-agnostic core: `mcp.go` previously decided whether to emit `WarnToolSchemaLossy` with `c.Provider.Name() != "google"`, and root itself held the Gemini-specific keyword knowledge (`unsupportedGeminiSchemaKeywords` / `collectUnsupportedSchemaKeywords` detecting `$ref`/`additionalProperties`/`oneOf`) — the same set the `google` adapter's `convertSchema` already strips, duplicated and able to drift. `Name()` is for labeling `Error.Provider` (D9), not behavioral dispatch.

```go
package agentkit

// ToolSchemaLimiter is an OPTIONAL capability interface a Provider may implement
// to declare the JSON-Schema keywords it will silently drop from a tool's input
// schema. The orchestrator discovers it by type assertion (like http.Flusher);
// a Provider that does not implement it is assumed to preserve full fidelity.
// This is the SPI seam for any provider-specific schema limitation — it carries
// no provider name and the core never special-cases a provider by identity.
type ToolSchemaLimiter interface {
    // UnsupportedSchemaKeywords returns, in deterministic (sorted) order, the
    // schema keywords this provider drops from the given JSON-Schema tool input.
    // Empty or nil means the provider preserves the schema as-is.
    UnsupportedSchemaKeywords(schema json.RawMessage) []string
}
```

Settled choices:
- **Optional interface, discovered by type assertion — the core knows no provider by name.** `mcpSchemaWarnings` becomes: `lim, ok := c.Provider.(ToolSchemaLimiter); if !ok { return nil }`, then for each MCP tool it calls `lim.UnsupportedSchemaKeywords(mt.schema)` and, when non-empty, assembles the warning. The provider-name string and the keyword-detection logic both leave root.
- **Root keeps warning *assembly*; the provider owns schema *knowledge*.** Root still owns `Warning`, the `WarnToolSchemaLossy` code, and the `mcpTool` `server`/`originalName` attribution, so it builds the `Warning{Setting:"mcp_schema", Code:WarnToolSchemaLossy, Detail: "<server>.<originalName> drops unsupported schema keywords for <provider>: <kw…>"}`. Only *which keywords are unsupported* moves to the provider — the one piece that is genuinely provider-specific.
- **`unsupportedGeminiSchemaKeywords` and `collectUnsupportedSchemaKeywords` move into the `google` adapter**, which implements `ToolSchemaLimiter` by reporting exactly the keyword set its `convertSchema` strips. One source of truth in the adapter; root↔google keyword drift is eliminated.
- **Timing and boundary semantics are unchanged.** Warnings are still produced at the `Send`/discovery boundary through the existing cached paths (`mcp.go` discovery and the `mcpToolCache` reuse), not deferred to a round-trip — D17's "schema lossiness surfaces at the boundary" promise holds. The only change is *how* the core learns the keyword set (ask the provider vs. hard-coded list + name check).
- **Anthropic/OpenAI/Z.ai do not implement `ToolSchemaLimiter`** (they accept the strict tool-name charset schemas without dropping these keywords), so they produce no schema warnings — now structurally (interface absent), not by name-exclusion.
- **D17 is rewritten in place** to describe schema-lossiness via `ToolSchemaLimiter` rather than a `Name()` check; **D9 notes** that providers may expose optional capability interfaces discovered by type assertion (the general pattern, of which this is the first instance). This Decision also gives `WarnToolSchemaLossy` its first Verification coverage — it previously carried no id and no test.

**Rejected.**
- *Status quo — `c.Provider.Name() != "google"`* — provider identity dispatched in the provider-agnostic core; silently fails to warn for any future schema-limited provider, and duplicates the Gemini keyword list root-side where it can drift from the adapter's `convertSchema`.
- *Have the provider's `RoundTrip` return the schema warnings (the macro-eval sketch)* — the adapter receives `[]Tool`, not the `mcpTool` wrapper, so it cannot attribute `server.originalName`; it would also re-emit the warning on every round-trip and shift the surfacing off the discovery boundary. The optional-interface seam is smaller and preserves attribution and timing.
- *A general `map[string]any` capability bag on the Provider* — untyped and undiscoverable; a named optional interface is the idiomatic Go expression and keeps the contract checkable.
- *A required method on the `Provider` SPI* — would force every provider (including those with no schema limits) to implement a no-op; optional via type assertion keeps the SPI minimal and the capability truly opt-in.

**Verification.**
- R-SKVI-TSZQ — a `Provider` implementing `ToolSchemaLimiter` whose `UnsupportedSchemaKeywords` returns a non-empty set for an attached MCP tool's schema causes a `WarnToolSchemaLossy` warning at the `Send`/discovery boundary that names those keywords and attributes the originating `<server>.<originalName>`; the warning is produced for a test provider **whose `Name()` is not `"google"`**, proving the core dispatches on the interface, not the provider name (a `Name()=="google"` implementation fails this).
- R-SNBB-LCH4 — a `Provider` that does **not** implement `ToolSchemaLimiter` produces no schema warnings for the same attached MCP tool (the warning slice is empty), and the orchestration core contains no branch on a provider name for this behavior.
- R-SOJ7-Z47T — the `google` adapter's `UnsupportedSchemaKeywords` returns exactly the keyword set it strips in `convertSchema` (`$ref`, `additionalProperties`, `oneOf`) for a tool schema that uses all three, in deterministic order, and the empty set for a schema using none — the adapter is the single source of that keyword knowledge.
