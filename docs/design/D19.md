# Decision 19 — The `EmbeddingProvider` SPI, package architecture & adapter-owned guarantees

**Decision.** A **separate** SPI — `EmbeddingProvider` — is the embeddings seam, parallel to the chat `Provider` SPI (D9) and exported for the same reason (sub-package adapters implement it across a package boundary). It is kept distinct from `Provider` because an embed call has no `System`/`Tools`/`History`/`Gen` and returns no streamed assistant message — sharing one interface would force meaningless methods on each side. Only the two embedding providers expose an embeddings constructor; the package graph and isolation rules of D9 are otherwise unchanged (root never imports a sub-package).

```go
package agentkit

// EmbeddingProvider is implemented by the embeddings-capable provider
// sub-packages (openai, google). Consumers obtain one from a sub-package's
// NewEmbedder and assign it to Embedder.Provider; they do not call Embed
// directly. This is the SPI for adding an embedding provider — distinct from
// the chat Provider SPI (D9) because embeddings share none of chat's request
// shape or streamed result.
type EmbeddingProvider interface {
    // Embed performs the whole embeddings request for one Embed call: it chunks
    // the batch across the provider's per-call limit, issues the round-trips,
    // L2-normalizes, and reassembles in input order. The orchestration-level
    // retry (D11) wraps each round-trip inside the adapter's chunk loop.
    Embed(ctx context.Context, req *EmbedRequest) *EmbedRoundTrip
    Name() string                                   // labels Error.Provider ("openai" | "google")
    Pricing(model string) (EmbeddingPricing, bool)  // registry lookup; false = unknown id (validity gate, D20)
}

// EmbedRequest is one Embed call's input, built by the Embedder from its fields.
// The adapter translates it to its wire format. Dimensions == 0 means native.
type EmbedRequest struct {
    Model      string
    Inputs     []string
    Role       InputType
    Dimensions int
    Retry      RetryPolicy
}

// EmbedRoundTrip is the assembled low-level result the Embedder lifts into a
// consumer EmbedResult. Vectors are already unit-normalized and in input order;
// Usage is summed across the request's chunks.
type EmbedRoundTrip struct { /* unexported */ }
func (r *EmbedRoundTrip) Vectors() [][]float32
func (r *EmbedRoundTrip) Usage() EmbeddingUsage
func (r *EmbedRoundTrip) Warnings() []Warning
func (r *EmbedRoundTrip) Err() error
```

Each embeddings-capable sub-package exposes a constructor typed to the embeddings SPI, parallel to its chat `New` (D5) and reusing the same `WithBaseURL`/`WithHTTPClient` options and the same per-provider credential injection. Only OpenAI and Google have one — embeddings are a deliberate subset (product):

```go
func openai.NewEmbedder(apiKey string, opts ...Option) agentkit.EmbeddingProvider // /v1/embeddings, bearer
func google.NewEmbedder(apiKey string, opts ...Option) agentkit.EmbeddingProvider // :batchEmbedContents, x-goog-api-key (Gemini API key, NOT Vertex/OAuth)
```

**Adapter implementation seam.** OpenAI's `/v1/embeddings` shape is shared with any future OpenAI-compatible embedding provider through an **`internal/openaicompat` embeddings variant** (parallel to the chat one, never public); Google's `embedContent`/`batchEmbedContents` shape, `task_type`, and normalization fix-up need their **own** adapter. Both reuse the existing rails verbatim: `internal/httpx` for the HTTP call, the D7 `Error` taxonomy + classification, and the D11 `RetryPolicy` executor.

Settled choices:
- **A distinct exported `EmbeddingProvider` SPI, not a widened `Provider`.** Embeddings carry none of `Request`'s `System`/`Messages`/`Tools`/`Gen` and return vectors, not a streamed `Message`; one interface would saddle each side with the other's dead methods. The two SPIs are siblings over the same provider sub-packages.
- **A parallel `NewEmbedder` constructor, leaving the chat `New` (D5) untouched.** The concrete client may share transport internals, but the *public* embeddings constructor is separately typed to `EmbeddingProvider` and exists only for the two embedding providers — so "OpenAI and Google only" is structural (no `anthropic.NewEmbedder`/`zai.NewEmbedder` exists to call), and D5's chat surface is not reopened.
- **The adapter owns the whole call, including chunking — there is no separate orchestrator.** Unlike chat (where a root orchestrator runs the tool loop above the per-round-trip adapter, D9/D10), embeddings have no loop: the adapter chunks the batch, issues each round-trip under the D11 retry policy, normalizes, and reassembles. The `Embedder.Embed` method is a thin lift of `EmbedRoundTrip` into `EmbedResult` plus cumulative accounting (D18) — the registry/cost math lives in D20.
- **Guarantee 1 — always L2-normalize, client-side, uniformly.** Every returned vector is unit-normalized by AgentKit regardless of provider or dimension. This is mandatory for `gemini-embedding-001`, which returns **un-normalized** vectors when the output is reduced below native (research §12.2), and is a cheap idempotent no-op when the provider already normalized (OpenAI, full-dim Google). The promise is *"AgentKit returns unit-normalized vectors,"* never *"the provider does"* (research §12.3).
- **Guarantee 2 — batch auto-chunk + order-preserving reassembly.** The adapter splits `Inputs` across the provider's per-call item ceiling (OpenAI up to 2048 items/call; Google's `batchEmbedContents` array limit), issues one round-trip per chunk, concatenates the results in input order, and sums `EmbeddingUsage` across chunks — so a batch sized for one provider does not break on another, and the consumer never manages limits (research §12.3).
- **Guarantee 3 — fail-loud on over-length input.** The Google adapter sends `autoTruncate:false` so an over-length input **errors** instead of silently truncating; OpenAI already returns a 400. Both map to the D7 **`ErrContextLength`** category. AgentKit never returns a vector for a silently-truncated input (research §12.3, the most dangerous divergence). AgentKit does **not** pre-count tokens client-side — the per-model `MaxInputTokens` in the registry (D20) is introspection metadata, not a client-side gate; the provider is the enforcer.
- **`InputType` lowering is per-adapter.** Google maps `InputQuery → RETRIEVAL_QUERY`, `InputDocument → RETRIEVAL_DOCUMENT`, and `InputUnspecified → omit task_type`; the OpenAI adapter ignores `InputType` entirely (symmetric embeddings, no task field on the wire). The neutral `InputType` never leaks a provider term.
- **Model validity gate mirrors D5.** At the `Embed` boundary the adapter rejects any `Model` its embeddings registry (D20) does not know with `ErrInvalidConfig`, before issuing a call — the same registry that prices the model and describes its capabilities, so supported ⇒ priced ⇒ describable is one structural guarantee.
- **Retry reuses D11 with no first-byte rule.** An embed round-trip is a non-streaming, idempotent request, so the streaming-idempotency carve-out (D11's no-retry-after-first-byte) does not apply: a retryable-category failure (`ErrRateLimited`/`ErrOverloaded`/`ErrServerError`/`ErrTimeout`/`ErrNetwork`) on any chunk's round-trip is retried under `Embedder.Retry` to exhaustion; non-retryable categories are never retried. Each chunk gets its own retry budget, exactly as each chat round-trip does.

**Rejected.**
- *One `Provider` interface covering both chat and embeddings* — forces `RoundTrip`-shaped methods onto embeddings and `Embed` onto chat-only providers (Anthropic, Z.ai); the sibling SPIs keep each provider implementing only what it offers.
- *Reusing the chat `New` to return a value that also embeds* — would make `anthropic.New(...)` structurally claim an embeddings capability it lacks; a separate `NewEmbedder` on only the two providers encodes the subset in the type system.
- *A root embeddings orchestrator layered above a per-chunk adapter (mirroring chat's D10)* — there is no tool loop or history to own; the only cross-chunk work is concatenation + usage summation, which belongs in the adapter next to the chunking that produced it.
- *Trusting providers to normalize* — Google does not for reduced dims; the only honest promise is library-side normalization.
- *Letting Google silently truncate (its default)* — corrupts retrieval quality with no signal; `autoTruncate:false` + `ErrContextLength` makes over-length uniform with OpenAI.
- *Client-side token counting to pre-reject over-length input* — duplicates the provider's own limit (which drifts), adds a tokenizer dependency per provider, and is redundant once the provider fails loud.

**Verification.**
- R-YGQZ-C8S2 — each embedding provider's `NewEmbedder` returns a value assignable to `Embedder.Provider`, and an `Embed` against it issues a correctly-authenticated request to that provider's embeddings endpoint (OpenAI `Authorization: Bearer` → `/v1/embeddings`; Google `x-goog-api-key` → `:batchEmbedContents`) and decodes one vector per input from the served fixture.
- R-YHYV-Q0IR — every vector returned from `Embed` is unit-length within float tolerance, for both providers and including a Google request with `Dimensions` set below native (which the provider returns un-normalized) — AgentKit's client-side L2-normalization, not the provider's.
- R-YJ6S-3S9G — a batch larger than the provider's per-call item limit is split into multiple round-trips and reassembled so the result has exactly `len(inputs)` vectors in input order, with `EmbeddingUsage` summed across the chunks (asserted against a fake server that records the chunk count).
- R-YKEO-HK05 — an over-length single input surfaces a provider `*Error` with `Category == ErrContextLength` (the Google adapter sending `autoTruncate:false` so the provider errors rather than truncating; OpenAI's 400 mapping likewise), and no silently-truncated vector is ever returned.
- R-YLMK-VBQU — the Google adapter lowers `InputQuery`/`InputDocument`/`InputUnspecified` to `task_type: RETRIEVAL_QUERY`/`RETRIEVAL_DOCUMENT`/omitted respectively on the wire, while the OpenAI adapter emits no task/role field for any `InputType` (asserted on the recorded request body).
- R-YMUH-93HJ — `Embed` with a `Model` id the selected provider's embeddings registry does not know surfaces `ErrInvalidConfig` (matchable via `errors.Is`) and issues no provider call (the validity gate resolves against the same registry that prices and describes the model, D20).
- R-YO2D-MV88 — a retryable-category failure on an embed round-trip (including on one chunk of a multi-chunk batch) is retried under `Embedder.Retry` per the D11 policy — with the injected clock asserting the attempt count — and a non-retryable-category failure is not retried; the no-retry-after-first-byte rule does not apply (embeddings are non-streaming and idempotent).
