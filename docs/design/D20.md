# Decision 20 — The embedding registry: usage, pricing/cost, and capability introspection

**Decision.** Embeddings get their **own** small usage and pricing types — `EmbeddingUsage` and a flat single-rate `EmbeddingPricing` — because the chat `Usage`/`Pricing` (D8/D16) carry output, reasoning, cache tiers, and context-length tiers that do not exist for embeddings (research §12.4). They reuse the root `Cost` type (D16) so dollar accounting is one shared currency. Each embedding provider sub-package ships one registry mapping a model id to its `EmbeddingPricing` **and** its `EmbeddingSpec` (native dimension, producible-dimension range, max input tokens), co-located with the exported model constant — the same single-source-of-truth pattern as the chat registry, so the validity gate (D19), the price, and the inspectable capabilities cannot drift apart. The supported set is closed and curated (product), so every supported model is priced **and** describable by construction.

```go
package agentkit

// EmbeddingUsage reports an embed call's token consumption. Embeddings bill
// input tokens only — no output, no reasoning, no cache tiers — so this is a
// deliberately smaller type than the chat Usage (D8). Total == InputTokens for
// every supported provider; it is kept as its own field for symmetry with chat
// and forward room. Summed across a batch's internal chunks (D19).
type EmbeddingUsage struct {
    InputTokens int64
    Total       int64 // == InputTokens
}

// EmbeddingPricing is one model's flat per-input-token rate in nano-USD (1e-9
// USD) — integer, so cost is exact with no float drift, exactly like chat
// Pricing (D16). No tiers and no output term: none of the supported embedding
// models tier by context length, so a single rate is both sufficient and exact.
type EmbeddingPricing struct {
    InputToken int64 // nano-USD per input token
}

// Cost computes an embed call's nano-USD cost: InputTokens · InputToken.
// Returns the shared root Cost type (D16), so Cost.USD() and the Embedder's
// cumulative TotalCost() (D18) work identically to chat.
func (p EmbeddingPricing) Cost(u EmbeddingUsage) Cost
```

```go
// EmbeddingSpec is the inspectable capability descriptor for one embedding
// model — the embeddings analogue of the reasoning ReasoningSpec (D16). It lets
// a consumer present choices and validate a requested dimension up front,
// without provider-specific knowledge. The producible-dimension range is what
// the D18 Dimensions gate consults: a requested d is producible iff
// MinDimension <= d <= MaxDimension (MaxDimension == NativeDimension).
type EmbeddingSpec struct {
    NativeDimension int // the default dimension when Embedder.Dimensions == 0
    MinDimension    int // inclusive lower bound on a producible target dimension
    MaxDimension    int // inclusive upper bound; equals NativeDimension
    MaxInputTokens  int // per-input token ceiling — introspection metadata, NOT a client-side gate (D19)
}

// EmbeddingInspector reads a provider's embedding-model capabilities. Implemented
// by a credential-blind package-level value in each embedding sub-package — NOT
// the EmbeddingProvider handle — so introspection needs no API key and no network
// (mirrors the ReasoningInspector split, D16).
type EmbeddingInspector interface {
    EmbeddingSpec(model string) (EmbeddingSpec, bool) // false if the model id is unknown
    SupportedEmbeddings() map[string]EmbeddingSpec    // every supported model's spec, for catalog rendering
}
```

```go
// package openai (and identically google): a credential-blind capability
// inspector value backed by the package's embedding registry — no handle, no
// network. Parallel to the chat `Reasoning` introspector (D16).
var Embeddings agentkit.EmbeddingInspector

// Exported model constants enumerate exactly the supported (and priced) set:
const (
    openai.EmbedModel3Small = "text-embedding-3-small"
    openai.EmbedModel3Large = "text-embedding-3-large"
    google.EmbedModelGemini001 = "gemini-embedding-001"

```

**Curated embedding-model set.** OpenAI `text-embedding-3-small` and `text-embedding-3-large`, and Google `gemini-embedding-001` — the GA, dollar-priced, dimension-flexible models verified in research §12.2. Legacy fixed-dimension models (`text-embedding-ada-002`, Google `embedding-001`/`text-embedding-004`) are **excluded** — ada-002 cannot shorten and the Google legacy ids are shut down (research §12.2/§12.7). Z.ai `embedding-3` is excluded with the rest of Z.ai embeddings (product; deferred pending a live smoke test, research §12.2).

**Baked-in embedding rate table** (nano-USD per input token; from research §12.4, official rates gathered 2026-06 — re-verify before release). Rate = USD-per-1M-tokens × 1000.

| Model | InputToken (nano-USD/tok) | USD / 1M tokens |
|---|---|---|
| text-embedding-3-small | 20 | $0.02 |
| text-embedding-3-large | 130 | $0.13 |
| gemini-embedding-001 | 150 | $0.15 |

**Baked-in capability table** (the `EmbeddingSpec` per model; from research §12.2 — re-verify before release).

| Model | NativeDimension | Min..Max producible | MaxInputTokens |
|---|---|---|---|
| text-embedding-3-small | 1536 | 1 .. 1536 | 8192 |
| text-embedding-3-large | 3072 | 1 .. 3072 | 8192 |
| gemini-embedding-001 | 3072 | 128 .. 3072 | 2048 |

Caveats the spec data must encode:
- **OpenAI v3 is Matryoshka** — `dimensions` may be reduced to any value at or below native (research §12.2), so the producible range is `1 .. native`. The exact accepted minimum is not documented; `1` is the permissive lower bound and the value is a re-verify-before-release item like the lower-confidence pricing rows.
- **Google `gemini-embedding-001` accepts `outputDimensionality` 128 .. 3072** (research §12.2); below native it returns un-normalized vectors, which D19's mandatory client-side normalization corrects.
- **`MaxInputTokens` is introspection metadata only** — D19 does not pre-count tokens; the provider enforces the limit and AgentKit surfaces `ErrContextLength` (the fail-loud guarantee). The field exists so a consumer can present/validate a choice, not to gate the call.

Settled choices:
- **A separate `EmbeddingUsage`/`EmbeddingPricing` pair, not the chat types.** Reusing `Usage`/`Pricing` would carry ~six always-zero fields (`Output`, `ReasoningOutput`, the three cache-write buckets, `CacheReadInput`) and a `(Output+ReasoningOutput)·rate` term that is structurally wrong for input-only billing (research §12.4). The small pair states exactly what embeddings bill.
- **Flat single-rate pricing, no `RateTier`.** None of the three supported models tiers by context length, so a single `InputToken` rate is exact; the tiered `RateTier` machinery (D16) earns its surface only on chat models that re-rate above a token threshold.
- **The root `Cost` type is shared.** `EmbeddingPricing.Cost` returns the same `Cost` (nano-USD integer, D16), so `Cost.USD()`, `EmbedResult.Cost()`, and `Embedder.TotalCost()` are one currency across both capabilities — the product's single dollar-cost promise.
- **No unpriced state — cost is always available.** A call produces usage only if the adapter accepted the model at the `Embed` boundary (D19); accepted ⇒ in the registry ⇒ priced. The public `Cost()`/`TotalCost()` return a bare `Cost`; the SPI keeps `Pricing(model) (EmbeddingPricing, bool)` as a structural validity check, and a runtime `false` for a call that ran is a failed invariant, never a silent zero.
- **One registry carries pricing + capabilities, exposed two ways.** Pricing reaches the adapter through the `EmbeddingProvider.Pricing` SPI (used in the call path); capabilities reach a consumer through the credential-blind package-level `Embeddings` inspector (static reads, no handle). "Supported ⇒ priced ⇒ describable" is therefore one structural guarantee, mirroring D16.
- **Capability introspection is credential-blind, mirroring D16's reasoning introspector.** `EmbeddingSpec`/`EmbeddingInspector` are root types (one shared shape); each sub-package exposes a package-level `Embeddings` value over its registry, so a consumer renders a model picker and validates a requested dimension with no API key and no client — a `map[string]EmbeddingInspector` keyed by provider name, no provider-specific branching.

**Rejected.**
- *Reusing chat `Usage`/`Pricing` for embeddings* — drags in output/reasoning/cache/tier fields that are always zero or meaningless and an output cost term that should not exist (research §12.4); a smaller honest pair is clearer and unliable to misprice.
- *`float64` USD* — float drift on accumulation; nano-USD integers are exact (as D16).
- *Tiered embedding pricing* — no supported embedding model tiers by context length; tiers would be dead structure.
- *Including legacy fixed-dimension models (ada-002, embedding-001)* — they cannot honor a requested dimension and the Google legacy ids are shut down; a curated set of dimension-flexible GA models keeps the Dimensions promise honest.
- *Capability introspection as an `EmbeddingProvider` handle method* — would force a model picker to construct a (fake-keyed) client for static metadata; the credential-blind package-level value avoids it, exactly as D16 argued for reasoning.
- *A root-level `agentkit.EmbeddingSpecFor(provider, model)`* — root cannot import sub-packages (D9 import-cycle rule); introspection must originate per sub-package.

**Verification.**
- R-YPAA-0MYX — for each provider, a recorded embeddings response maps to `EmbeddingUsage` (OpenAI `usage.prompt_tokens`; Google `usageMetadata.promptTokenCount`) with `Total == InputTokens`, and a multi-chunk batch's usage is the sum across its chunks.
- R-YQI6-EEPM — `EmbeddingPricing.Cost(u)` equals `u.InputTokens × InputToken` in integer nano-USD; `EmbedResult.Cost()` and `Embedder.TotalCost()` derive from it, and cost is available for every supported model (no unpriced state).
- R-YRQ2-S6GB — each sub-package's `Embeddings` inspector returns specs without constructing an `EmbeddingProvider` handle or issuing any network call (credential-blind: callable with no API key set).
- R-YSXZ-5Y70 — `EmbeddingSpec(model)` returns `(spec, true)` for a supported model id and `(_, false)` for an unknown one, and `SupportedEmbeddings()` is keyed by exactly the exported model constants.
- R-YU5V-JPXP — each provider registry's `EmbeddingPricing.InputToken` equals the rate in this decision's rate table, so shipped code and doc cannot silently diverge (golden).
- R-YVDR-XHOE — each provider registry's `EmbeddingSpec` per model (NativeDimension, Min/MaxDimension, MaxInputTokens) equals this decision's capability table, so shipped code and doc cannot diverge on a model's native dimension, producible range, or input ceiling (golden).
- R-YWLO-B9F3 — every exported embedding-model constant in every embedding sub-package resolves to **both** an `EmbeddingPricing` and an `EmbeddingSpec` in its registry (supported ⇒ priced ⇒ describable; the registry is complete, so no call can run unpriced or undescribable).
