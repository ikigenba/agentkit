# Phase 18 — Fix: shared `internal/openaicompat` replayed tool-call `arguments` serialized as object, not string
*Realizes design Decision 9 (the Chat-Completions `tool_calls.arguments` serialization correction) and Decision 12 (`internal/openaicompat`). Depends on Phase 10; the Chat-Completions sibling of the Phase 17 OpenAI-Responses defect.*

Fix the same outbound/inbound `arguments` asymmetry as Phase 17, but in the shared Chat-Completions adapter (`internal/openaicompat/openaicompat.go`) that backs the public `zai` package (and any future OpenAI-compatible provider). A replayed assistant tool call serializes its `tool_calls[].function.arguments` as a nested JSON **object** instead of the JSON **string** the OpenAI Chat-Completions wire requires: the request-side field `toolFunction.Arguments` is typed `json.RawMessage` and set via `cloneRaw(b.Input)` in `convertMessage`, so a `ToolUseBlock.Input` of `{"command":"ls -1 | wc -l"}` ships as the object `{"command":"…"}` rather than the string `"{\"command\":\"…\"}"`. The adapter's *parse* side already types this field `string` (`toolFunctionDelta.Arguments`), confirming the asymmetry. The first round-trip's tool calls succeed (model → tool execution), but the **follow-up round-trip of the same `Send`** — which resends the assistant `ToolUseBlock`s as `tool_calls` to fetch the final answer — is the first to serialize `arguments` outbound. This was masked because Z.ai's default base URL (`api/paas/v4`) tolerates the object form; it surfaces only once `zai.base_url` points at the strict coding endpoint (`api/coding/paas/v4`), which rejects it with `400 Invalid API parameter (type=1210)`. Observed via agentrepl (`-c provider=zai -c zai.base_url=https://api.z.ai/api/coding/paas/v4`): a `bash` tool call runs, then the turn errors before the final answer. End state: a replayed `tool_calls[].function.arguments` is a JSON string equal to the canonical `ToolUseBlock.Input` bytes (the outbound field retyped `string`, set to `string(b.Input)`, with an empty/invalid input guarded to `"{}"` to mirror the inbound assembly), the inbound parse unchanged. The existing `zai` tool-turn test asserts the replayed tool output and reasoning are *present* but never the *type* of `arguments` — that gap is closed by a replay/wire test asserting the second request body's assistant `function.arguments` is a JSON string. (Chat-Completions-specific within the OpenAI family; Anthropic/Gemini take structured tool input directly and are unaffected.)

**Done when:** R-ZCMP-ARG8 is covered by a clearly-named test (Chat-Completions/`zai` replay exercising the tool follow-up round-trip, asserting `tool_calls[].function.arguments` is a JSON string and no `400`/`ErrInvalidRequest`/`type=1210`); suite green.
