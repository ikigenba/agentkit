# Phase 20 — Fix: Google adapter nests the replayed `thoughtSignature` inside `functionCall` instead of on the part
*Realizes design Decision 9 (the Gemini request-build correction). Depends on Phase 11; the outbound replay sibling of Phase 19's inbound parse fix, and unmasked by it.*

Fix a defect in the `google` adapter's request build (`google/google.go`, `partsFromMessage`): when replaying an assistant `ToolUseBlock` that carries a bound `thoughtSignature`, the adapter writes the signature **into** the `functionCall` map (`call["thoughtSignature"] = sig`) instead of onto the part wrapper that holds the call. Gemini's wire schema places `thoughtSignature` at the **part** level — a sibling of `functionCall`/`text`/`functionResponse` — and the `functionCall` object accepts only `name`, `args`, and `id`. The adapter's own *parse* side already models this correctly (the `part` struct carries `thoughtSignature` as a sibling of `FunctionCall`, whose struct has only `Name`/`Args`), so this is a pure outbound/inbound asymmetry — the same shape as the Phase 17/18 `arguments` traps, here on the part-vs-object axis. Gemini rejects the misplaced field with `400 INVALID_ARGUMENT: Invalid JSON payload received. Unknown name "thoughtSignature" at 'contents[N].parts[0].function_call': Cannot find field.` This is **unmasked by Phase 19**: before that fix a signature on a live `functionCall` part was dropped on parse, so no `ReasoningBlock.Opaque` survived to replay and `partsFromMessage` never emitted a `thoughtSignature`; once Phase 19 began capturing the signature into `Opaque` (bound to its `ToolUseBlock` via `BoundToID`), the next round-trip became the first to serialize it outbound — in the wrong location. Like the OpenAI replay traps it bites only on **replay**: the model's first tool-call round-trip succeeds (nothing replayed yet), and the follow-up round-trip that resends the assistant `ToolUseBlock` is the first to ship the misplaced signature. Observed via agentrepl (`-c provider=google -c model=gemini-2.5-flash`): a clean first text turn, the model then emits a `read` tool call (now assembled, post-Phase-19), the tool runs and returns its result, and the turn that replays that tool call fails with the `Unknown name "thoughtSignature" ... function_call` 400 at `contents[3]` (the second assistant turn) before any further output. End state: the replayed signature is emitted as a **part-level** key — `{"functionCall":{"name":…,"args":…,"id":…}, "thoughtSignature":<sig>}` — with the `functionCall` object carrying only `name`/`args`/`id`, mirroring the parse-side `part` shape; a `ToolUseBlock` with no bound signature emits a bare `functionCall` part as before. A wire/replay test exercising a tool-turn follow-up round-trip pins the regression: it asserts the rebuilt request body places `thoughtSignature` on the part (not inside `function_call`) and that the round-trip does not surface `400`/`ErrInvalidRequest`. (Gemini-specific — Anthropic/OpenAI/Z.ai have no part-level signature field and are unaffected.)

**Done when:** R-GSIG-PT07 is covered by a clearly-named test (Google wire/replay exercising a tool-turn follow-up round-trip, asserting the replayed `thoughtSignature` is a part-level sibling of `functionCall` — the `functionCall` object holding only `name`/`args`/`id` — and no `400`/`ErrInvalidRequest`); suite green.
