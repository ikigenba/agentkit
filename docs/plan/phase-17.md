# Phase 17 — Fix: OpenAI replayed tool-call `arguments` serialized as object, not string
*Realizes design Decision 9 (the OpenAI Responses `function_call` serialization correction). Depends on Phase 9; unmasked by Phase 16 (whose `summary` fix let request validation reach the `arguments` field).*

Fix a defect in the `openai` adapter's request build (`openai/openai.go`): a replayed assistant tool call serializes its `arguments` as a nested JSON **object** instead of the JSON **string** OpenAI's Responses API requires. The `function_call` input item's `arguments` field is built from `inputItem.Arguments`, typed `json.RawMessage` and set via `cloneRaw(block.Input)`; a `json.RawMessage` marshals verbatim, so `ToolUseBlock.Input` of `{"path":"PING"}` ships as the object `{"path":"PING"}` rather than the string `"{\"path\":\"PING\"}"`. The adapter's *parse* side already types this field `string` (`outputItem.Arguments`), so the bug is an outbound/inbound asymmetry. The first round-trip's tool calls succeed (model → tool execution), but the **follow-up round-trip of the same `Send`** — which resends the assistant `ToolUseBlock`s as `function_call` items to fetch the model's final answer — is the first to serialize `arguments` outbound and fails with `400 Invalid type for 'input[N].arguments': expected a string, but got an object instead` (the replayed call lands at `input[1]`). This was masked until the Phase 16 `summary` fix, which previously tripped `400` earlier in the same request. Observed via agentrepl (`-c provider=openai`): a multi-tool turn runs both tools, then errors before the final answer. End state: a replayed `function_call`'s `arguments` is a JSON string equal to the canonical `ToolUseBlock.Input` bytes (the outbound field retyped `string`, set to `string(block.Input)`, `omitempty` retained), with the inbound parse unchanged. A replay test exercising a multi-tool turn's follow-up round-trip pins the regression: it asserts the request body's `function_call.arguments` is a JSON string and that the round-trip does not surface `400`/`ErrInvalidRequest`. (OpenAI-Responses-specific — Anthropic/Gemini take structured tool input directly and are unaffected.)

**Done when:** R-UJNS-PFLL is covered by a clearly-named test (OpenAI replay exercising the multi-tool follow-up round-trip, asserting `function_call.arguments` is a JSON string and no `400`/`ErrInvalidRequest`); suite green.
